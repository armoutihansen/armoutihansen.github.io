# Premium-dark redesign anchored on one interactive quant hero

## Status

Accepted. Supersedes [0001](0001-scholar-coder-stripped-redesign.md) and
[0002](0002-editorial-brief-redesign.md).

## Context

0001 (scholar-coder, strip-to-bones) and 0002 (Editorial Brief, ivory + teal serif)
were both deliberately restrained. Reviewed against the actual goal — a site that reads as
*impressive* to a quantitative-analyst hiring audience — restraint underperformed: it read
as a quiet academic page, which is the opposite signal from the developer/design-engineer
sites the owner found compelling.

The direction here was settled through a structured grilling of the trade-offs (see commit
history), not a re-skin. The hard constraint throughout: everything on the page must be
grounded in demonstrable ability (CV, research, GitHub) — no overclaiming.

## Decision

1. **Audience & takeaway.** Optimise for one viewer — a quantitative-analyst hiring lead
   with ~20 seconds — who must come away thinking *"rigorous, modern, actually builds
   things,"* never *"front-end designer."* Every choice is scored against that reaction.
2. **One load-bearing centrepiece, no flex tricks.** The wow is carried by a single
   substantive element, not by developer ornament (no cursor-spotlight, no dark-as-flex,
   no motion for its own sake). The shell stays restrained-premium; the hero does the work.
3. **The hero is a two-panel deck** the visitor scrolls/tabs between — both are genuine
   analyses of the owner's, labelled so a visitor sees there are two:
   - **Model confidence → decision**: an interactive view of a real logistic-regression
     classifier's predicted probabilities over held-out cases; dragging a confidence band
     routes the uncertain middle to review and shows the coverage/accuracy trade-off. This
     mirrors the owner's AXA model-evaluation work (confidence, thresholds, human review).
   - **Risk per trip → time of day**: the real finding from the CitiBike data-science
     project (collision severity per trip, BallTree station-matching, EB smoothing) — risk
     peaks at night, ~4× daytime.
4. **Shell: dark-first premium.** Warm-charcoal paper, a single **amber** accent (no teal),
   the existing `[data-theme]` light mode retained but secondary. Achromatic everywhere
   except the one accent.
5. **Typography.** Hanken Grotesk for body and headings; **JetBrains Mono with tabular
   figures** as the "quant tell" (every number is monospaced); **Fraunces reserved for the
   name only** — the single serif moment. IBM Plex Sans and Newsreader are dropped.
6. **Real, precomputed data — never a runtime model.** The hero classifier is generated by
   `scripts/gen_hero_data.py` → `src/data/hero-model.json`; the GitHub footprint by
   `scripts/gen_github_data.mjs` → `src/data/github.json`. Both are committed and shipped
   static; the site makes no model or API calls at build or request time.
7. **Breadth is distributed, not stacked into the hero.** Programming/ML projects are
   foregrounded in the home's featured set (RAG → CitiBike → Economic Theories & ML); the
   open-source footprint lives as a restrained strip on the Work page, not a hero slide.
8. **Positioning.** "Quantitative analyst" in the broad sense (statistics/econometrics +
   programming + ML); researcher is a secondary facet. AXA is named (owner's decision).

## Considered alternatives

- **Keep 0001/0002 minimalism.** Rejected: the owner found it unimpressive for the audience.
- **Spectacle (WebGL/3D/game-like creative-dev sites).** Rejected: signals *creative
  front-end developer*, is off-identity for a quant, and is high build-and-maintain cost.
- **A four-panel hero carousel** (model + RAG + neural-net + GitHub stats behind dots).
  Rejected: dilutes the single statement; carousel slides past the first go unseen; a
  neural-net panel would overclaim (the owner uses pretrained embeddings, not trained nets);
  a GitHub language donut is a generic-dev tell. Resolved to a **two-panel deck of genuine
  analyses** plus distributed breadth.

## Consequences

- The hero is real client-side code (`src/components/Hero.astro`, Canvas 2D). It must
  degrade gracefully — server-rendered readouts, a `prefers-reduced-motion` static frame,
  and keyboard control on the band — and that contract should be preserved.
- Two data snapshots (`hero-model.json`, `github.json`) are regenerated by re-running their
  scripts; they will drift if never refreshed (the GitHub "last active" date especially).
- A future contributor adding a third/fourth hero panel, a GitHub language donut,
  dark-as-flex ornament, or a decorative chart not tied to real work is fighting this ADR —
  update it rather than diverge silently.
- The honesty rule is load-bearing: every visualisation must depict a method or finding the
  owner can defend in an interview. Approximate values (e.g. CitiBike bars read off the
  project's own chart) are acceptable; invented results are not.

## Amendments

### 2026-06-22 — Panel 2 becomes the second interactive analysis (exposure normalization)

Decision 3 originally specified Panel 2 as a *static* "risk per trip → time of day" bar
chart, in service of decision 2's "one load-bearing centrepiece, no motion for its own
sake." Reviewing the rendered hero, Panel 2 was the weakest element — a generic bar chart
of five **eyeballed** values, the least data-backed thing in a hero whose whole premise is
real precomputed data.

Panel 2 is now interactive: a toggle between **total crash severity** (peaks midday) and
**risk per trip** (severity ÷ trip exposure, EB-smoothed; peaks at night ≈3.7×). Dividing
by exposure **inverts the ranking** — night has the fewest crashes and fewest riders but
the highest per-trip risk. That flip *is* the interaction.

This deliberately relaxes "one interactive centrepiece" to **two interactive analyses**,
justified because the second one **teaches a distinct idea** (the actuarially-credible
per-trip risk an insurer prices on — on-identity) rather than adding motion. It stays the
*supporting act*: a two-state toggle, not a drag-to-explore showpiece competing with Panel
1. Decision 2's real target — "no motion for its own sake" — is preserved; the motion
carries the insight.

It also **strengthens the honesty rule**: the eyeballed bars are replaced by real values
recomputed from the owner's CitiBike project (`scripts/gen_citibike_tod.py` →
`scripts/data/citibike_tod_risk.json`; provenance: AXA/DSC `notebooks/risk_analysis.ipynb`).
The "approximate values… read off the project's own chart" clause above no longer applies
to Panel 2 — it now ships committed, regenerable data like Panel 1.
